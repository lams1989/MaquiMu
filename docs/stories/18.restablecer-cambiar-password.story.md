# Historia #18: Restablecer y Cambiar Contraseña

**Estado:** Pendiente 🔲
**Fecha Análisis Arquitectónico:** 2026-02-24
**Origen:** Solicitud Directa (Seguridad / Gestión de Acceso)
**Estimación:** 8 Puntos

---

**Como:** Usuario del sistema (Cliente u Operario)
**Quiero:** Poder cambiar mi contraseña desde mi perfil, y si la pierdo, solicitar un restablecimiento que el operario gestione.
**Para:** Mantener la seguridad de mi cuenta y poder recuperar el acceso si olvido mi contraseña.

**Criterios de Aceptación:**

## Funcionalidad A — Cambio de Contraseña (Auto-servicio)

1.  **Acceso desde el navbar:**
    *   Al hacer click en el **nombre de usuario** en el navbar (pill de usuario), se abre un menú/dropdown con la opción **"Cambiar Contraseña"**.
    *   Aplica tanto para rol OPERARIO como CLIENTE.

2.  **Modal de cambio de contraseña:**
    *   Se abre un modal con los campos:
        *   Contraseña actual (requerida, para verificación)
        *   Nueva contraseña (mín. 6 caracteres)
        *   Confirmar nueva contraseña (debe coincidir)
    *   Validaciones: contraseña actual correcta, nueva ≠ actual, confirmación coincide.
    *   Al guardar: la contraseña se actualiza en la BD con BCrypt hash.
    *   Mensaje de éxito: _"Contraseña actualizada correctamente."_

## Funcionalidad B — "Olvidé mi Contraseña" (Gestión por Operario)

3.  **Botón en login:**
    *   En la página de login (`/auth/login`), se agrega un enlace/botón: **"¿Olvidaste tu contraseña?"**.
    *   Al hacer click, se abre un modal pidiendo el **email** registrado.
    *   El sistema valida que el email exista en la BD.
    *   Si existe, se crea una solicitud de restablecimiento con estado `RESTABLECER` asociada al usuario.
    *   El usuario ve: _"Se ha enviado una solicitud de restablecimiento. Un operario te asignará una contraseña temporal en un plazo de 1 a 3 días hábiles."_

4.  **Estado RESTABLECER en el módulo de clientes (Operario):**
    *   En `/admin/clients`, el operario ve un filtro/badge **"Solicitudes de Restablecimiento"** con usuarios en estado `RESTABLECER`.
    *   Para cada solicitud, el operario ve: nombre del cliente, email, fecha de solicitud.

5.  **Modal de asignación de contraseña temporal (Operario):**
    *   El operario hace click en **"Asignar Contraseña"** en la solicitud.
    *   Se abre un modal con:
        *   Información del usuario (nombre, email, solo lectura).
        *   Campo para ingresar una **contraseña temporal** (mín. 6 caracteres) o botón **"Generar Automáticamente"** que crea una contraseña aleatoria visible.
        *   Confirmar contraseña temporal.
    *   Al guardar:
        *   La contraseña del usuario se actualiza en BD (BCrypt hash).
        *   Se envía un **correo electrónico** al usuario con la contraseña temporal.
        *   El estado del usuario vuelve a `ACTIVO`.
        *   La solicitud de restablecimiento se marca como resuelta.
    *   Mensaje de éxito: _"Contraseña temporal asignada y enviada al correo {email}."_

6.  **Notificación al operario:**
    *   Cuando un cliente solicita restablecimiento, el operario recibe notificación en la campana: _"Solicitud de restablecimiento de contraseña: {nombre} ({email})"_.
    *   Al hacer click, navega a `/admin/clients` con el filtro de restablecimiento.

7.  **Primer login con contraseña temporal:**
    *   Cuando el usuario ingresa con la contraseña temporal, puede hacer login normalmente.
    *   (Opcional/Futuro: forzar cambio de contraseña en el primer login con contraseña temporal.)

## Análisis Arquitectónico

### Diagnóstico del Código Actual

#### Modelo `Usuario`
**Archivo:** `dominio/.../autenticacion/modelo/Usuario.java`

**Problema:** No tiene campo `estado` (se agrega en HU-16 como `EstadoUsuario`). Para esta HU se necesita agregar el valor `RESTABLECER` al enum `EstadoUsuario`.

**OK:** Se reutiliza para hashear la nueva contraseña / contraseña temporal.

#### `ManejadorAutenticarUsuario`
- Usa Spring Security `AuthenticationManager.authenticate()`.
- **No valida** si la contraseña es temporal.

#### Frontend — `login.component.html`
- No tiene enlace/botón "Olvidé mi contraseña".

#### Frontend — Navbars (admin + client)
- Muestran nombre y avatar del usuario.
- No tienen dropdown con opción "Cambiar Contraseña".

#### Infraestructura — Envío de correo
- **No existe** configuración de email/SMTP en el proyecto actual.
- Se necesita agregar Spring Boot Starter Mail + configuración SMTP.

### Decisiones de Diseño

**Patrón:** Dos flujos independientes que comparten infraestructura de contraseñas:
1. **Cambio de contraseña (self-service):** El usuario autenticado cambia su propia contraseña → nuevo endpoint protegido.
2. **Restablecimiento (operario-mediated):** El cliente solicita → el operario asigna temporal → envío por email.

**Envío de correo:** Spring Boot Starter Mail con JavaMailSender (SMTP configurable via `application.properties`).

**Dependencia con HU-16:** Esta HU asume que el enum `EstadoUsuario` ya existe (creado en HU-16). Se agrega el valor `RESTABLECER`. Si HU-16 no se implementa primero, se creará el enum aquí.

### Flujos de Interacción

#### Flujo A — Cambio de Contraseña (Self-Service)

```
[Usuario logueado] → Click en nombre (navbar)
    → Dropdown → "Cambiar Contraseña"
        → Modal: contraseña actual + nueva + confirmar
            → PATCH /usuarios/cambiar-password
                → 200 OK → "Contraseña actualizada"
                → 400 → "Contraseña actual incorrecta"
```

#### Flujo B — Olvidé mi Contraseña (Mediado por Operario)

```
[Cliente en /auth/login] → Click "¿Olvidaste tu contraseña?"
    → Modal: ingresa email
        → POST /auth/solicitar-restablecimiento
            → 200 → "Solicitud enviada, espera 1-3 días"
            → 404 → "Email no registrado"

[Operario ve notificación] → Click campana
    → Navega a /admin/clients → Tab "Restablecimiento"
        → Click "Asignar Contraseña"
            → Modal: contraseña temporal (manual o auto-generar)
                → PATCH /usuarios/{id}/asignar-password
                    → Backend: hashea, actualiza BD, envía email
                    → 200 → "Contraseña enviada a {email}"

[Cliente recibe email] → Ingresa con contraseña temporal → Login exitoso
```

### Máquina de Estados del Usuario (Actualizada — incluye HU-16)

```
[REGISTRO]  →  PENDIENTE_APROBACION  →  ACTIVO  ←→  RESTABLECER
                       ↓                    ↑           ↓
                   RECHAZADO                └── ACTIVO (password resuelto)
```

| Estado | Login Permitido | Cambiable desde | Acción |
|--------|----------------|-----------------|--------|
| `PENDIENTE_APROBACION` | ❌ | Registro (CLIENTE) | Operario aprueba/rechaza |
| `ACTIVO` | ✅ | Aprobación / Resolución | Normal |
| `RECHAZADO` | ❌ | Rechazo de aprobación | Contactar admin |
| `RESTABLECER` | ❌ | Solicitud de restablecimiento | Operario asigna contraseña temporal |

### Notas Técnicas

- **Generación de contraseña temporal:** Se recomienda una función que genera una cadena alfanumérica de 8 caracteres (ej: `Ab3xK9mP`). Visible en el modal del operario antes de guardar.
- **Seguridad:** El endpoint `POST /auth/solicitar-restablecimiento` es público pero debe tener rate-limiting básico para evitar abuso (ej: máx. 3 solicitudes por email en 24h). Para la primera fase, se puede omitir el rate-limiting.
- **Plantilla de correo:** Un HTML simple con el logo de MaquiMu, mensaje de bienvenida, la contraseña temporal en texto visible, e instrucciones de cambiarla al ingresar.
- **Login durante estado RESTABLECER:** El usuario **NO puede** hacer login mientras está en este estado. Solo cuando el operario asigne la contraseña temporal y el estado vuelva a ACTIVO podrá ingresar.
