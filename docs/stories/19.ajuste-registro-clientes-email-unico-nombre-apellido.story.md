# Historia #19: Ajuste de Registro de Clientes (Email Único + Nombre/Razón y Apellido)

**Estado:** Finalizada ✅
**Fecha Creación:** 2026-02-24
**Fecha Análisis Arquitectónico:** 2026-02-24
**Fecha Refinamiento:** 2026-02-24
**Fecha Implementación:** 2026-02-24
**Origen:** Solicitud directa de negocio
**Estimación:** 3 Puntos

---

**Como:** Operario del sistema
**Quiero:** Que el registro/edición de clientes no permita emails duplicados y permita gestionar nombre o razón social junto con apellido.
**Para:** Mantener integridad de datos y una captura de información de cliente más completa.

**Criterios de Aceptación:**
- El sistema no permite crear cliente con `email` duplicado.
- El sistema no permite actualizar un cliente a un `email` ya usado por otro cliente.
- La entidad cliente soporta `nombreCliente` (nombre o razón social) y `apellido`.
- El frontend de Gestión de Clientes permite capturar y visualizar `apellido`.
- Persona natural exige `apellido`; persona jurídica lo deja opcional.
- La base de datos incorpora columna `apellido` y restricción única para `email`.
- El manejo de errores mantiene el mensaje de backend en UI (alerta roja en modal).

---

## Análisis Arquitectónico

### Decisiones
- Se mantiene `nombreCliente` como campo principal (nombre o razón social) para compatibilidad.
- Se agrega `apellido` como nuevo campo opcional en modelo y BD.
- La unicidad de email se implementa en doble defensa:
  - Regla de aplicación (`ClienteDao` + manejadores).
  - Constraint en BD (`UNIQUE` en `clientes.email`).

### Impacto por Capa
- **Dominio:** `Cliente` agrega `apellido` y método `getNombreCompleto()`.
- **Aplicación:** comandos/fábrica/manejadores de cliente actualizados para `apellido` y validación email único.
- **Infraestructura Backend:** entidad JPA, repositorios y DAO con búsqueda/existencia por email.
- **Base de Datos:** migración Flyway nueva + actualización de esquema base.
- **Frontend:** modelo TS, modal de clientes y tabla de clientes actualizados para `apellido`.
