## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** CQRS (Query Side) - Agregación de Datos con Seguridad

**Justificación:**
*   **Seguridad:** Filtrado estricto de datos por `clienteId` para garantizar privacidad.
*   **Eficiencia:** Consultas `COUNT` optimizadas en BD en lugar de cargar listas completas.
*   **Consistencia:** Sigue el mismo patrón de agregación que el Dashboard de Operario (HU 11).

**Componentes Principales:**

*   **Aplicación (Consulta):**
    *   `ConsultaDashboardCliente`: DTO con `clienteId`.
    *   `ManejadorDashboardCliente`: Orquestador que invoca los DAOs filtrados.
    *   `DashboardClienteDTO`: Objeto de transferencia con los KPIs.

*   **Dominio (Puertos):**
    *   `AlquilerDao`: Añadir `long contarPorClienteYEstado(Long clienteId, EstadoAlquiler)`.
    *   `FacturaDao`: Añadir `long contarPorClienteYEstado(Long clienteId, EstadoPago)`.

*   **Infraestructura:**
    *   `ConsultaControladorDashboard`: Añadir endpoint para cliente.
    *   Repositorios JPA implementando los conteos filtrados (`countByClienteIdAndEstado`).

### Especificaciones Técnicas

**API Endpoint:**
*   `GET /api/maquimu/v1/dashboard/cliente`
*   **Respuesta:**
    ```json
    {
      "alquileresActivos": 2,
      "facturasPendientes": 1
    }
    ```

**Integraciones:**
*   El manejador agrega datos de 2 fuentes distintas (Alquileres y Facturas).
*   Requiere extracción del `clienteId` del token JWT.
