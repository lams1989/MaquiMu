# Historia #16: Aprobación de Registro de Cliente por Operario

**Estado:** Pendiente 🔲
**Fecha Análisis Arquitectónico:** 2026-02-24
**Origen:** Solicitud Directa (Seguridad / Flujo de Control)
**Estimación:** 8 Puntos

---

**Como:** Operario del sistema
**Quiero:** Aprobar o rechazar el registro de nuevos usuarios con rol Cliente antes de que puedan acceder al sistema.
**Para:** Tener control sobre quién utiliza la plataforma y evitar registros fraudulentos, con un plazo de validación de 1 a 3 días hábiles.

**Criterios de Aceptación:**

1.  **Registro crea cuenta pendiente:**
    *   Cuando un usuario se registra como CLIENTE desde `/auth/register`, su cuenta se crea con estado `PENDIENTE_APROBACION`.
    *   El usuario ve un mensaje: _"Tu cuenta ha sido creada exitosamente. Un operario la revisará en un plazo de 1 a 3 días hábiles. Te notificaremos cuando sea aprobada."_
    *   El usuario **NO puede iniciar sesión** mientras su cuenta esté en estado `PENDIENTE_APROBACION`.

2.  **Login denegado para cuentas pendientes:**
    *   Si un usuario con estado `PENDIENTE_APROBACION` intenta hacer login, el sistema muestra: _"Tu cuenta aún está pendiente de aprobación. Por favor espera de 1 a 3 días hábiles."_
    *   Si un usuario con estado `RECHAZADO` intenta hacer login, muestra: _"Tu cuenta ha sido rechazada. Contacta al administrador para más información."_

3.  **Panel de gestión para el operario:**
    *   En el módulo de Clientes (`/admin/clients`), el operario ve una pestaña o filtro **"Pendientes de Aprobación"** con los registros en estado `PENDIENTE_APROBACION`.
    *   El operario puede ver los datos del cliente (nombre, identificación, email, fecha de registro).
    *   Puede **Aprobar** → el estado del usuario cambia a `ACTIVO` y el cliente puede hacer login.
    *   Puede **Rechazar** con un motivo → el estado del usuario cambia a `RECHAZADO` y el cliente no puede ingresar.

4.  **Notificación al operario:**
    *   Cuando un nuevo cliente se registra, el operario recibe una notificación en la campana: _"Nuevo registro de cliente pendiente de aprobación: {nombre}"_.
    *   Al hacer click, navega a `/admin/clients` con el filtro de pendientes.

5.  **Botón de retorno al login:**
    *   En el formulario de registro (`/auth/register`) se agrega un botón visible para **"Volver al Login"** que navega a `/auth/login`.

6.  **Seguridad (Restricción de auto-registro de Operario):**
    *   El formulario de registro **solo permite el rol CLIENTE**. El select de "Tipo de Usuario" se elimina (solo clientes se auto-registran).
    *   Los operarios solo pueden ser creados directamente en base de datos o por otro operario (futuro).

## Análisis Arquitectónico

### Diagnóstico del Código Actual

#### Backend — Modelo de dominio `Usuario`
**Archivo:** `dominio/.../autenticacion/modelo/Usuario.java`
```java
public class Usuario {
    private Long id;
    private String nombreCompleto;
    private String email;
    private String passwordHash;
    private RolUsuario rol;          // OPERARIO | CLIENTE
    private LocalDateTime fechaCreacion;
    private Long clienteId;
}
```
**Problema:** No tiene campo `estado`. Actualmente cualquier usuario registrado puede hacer login inmediatamente.

#### Backend — `ManejadorRegistrarUsuario`
**Archivo:** `aplicacion/.../autenticacion/comando/manejador/ManejadorRegistrarUsuario.java`
- Registra al usuario y al cliente en una sola transacción.
- No asigna ningún estado al usuario — asume que está activo desde el momento de creación.

#### Backend — `ManejadorAutenticarUsuario`
**Archivo:** `aplicacion/.../autenticacion/consulta/manejador/ManejadorAutenticarUsuario.java`
- Autentica con Spring Security `AuthenticationManager`.
- **No valida el estado del usuario** — permite login a todos los usuarios con credenciales válidas.

#### Backend — Tabla `usuarios` (DB)
```sql
CREATE TABLE usuarios (
  usuario_id BIGINT AUTO_INCREMENT PRIMARY KEY,
  nombre_completo VARCHAR(255) NOT NULL,
  email VARCHAR(100) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  rol ENUM('OPERARIO', 'CLIENTE') NOT NULL,
  fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
**Problema:** No existe columna `estado_cuenta`.

#### Frontend — `register.component.html`
- Tiene un `<select>` que permite elegir entre CLIENTE y OPERARIO (vulnerabilidad).
- No tiene botón de "Volver al Login" (solo un link pequeño al final).
- Redirige al login tras registro exitoso sin mensaje de "espera aprobación".

#### Frontend — `login.component.html`
- No maneja un estado intermedio de "cuenta pendiente".

#### Frontend — `notification.service.ts`
- Sistema de polling cada 30s con `BehaviorSubject`.
- No detecta nuevos registros de clientes (solo cambios en alquileres y facturas).

---

### Decisiones de Diseño

**Patrón:** Extensión del modelo de dominio + CQRS (nuevos comandos para aprobar/rechazar)

**Justificación:**
- Se agrega un **enum** `EstadoUsuario` (`PENDIENTE_APROBACION`, `ACTIVO`, `RECHAZADO`) al dominio.
- Se valida el estado en el flujo de login (dominio/aplicación, no solo en seguridad).
- Se reutiliza la infraestructura de notificaciones existente para alertar al operario.

### Máquina de Estados del Usuario

```
[REGISTRO] → PENDIENTE_APROBACION → ACTIVO   (operario aprueba)
                                   → RECHAZADO (operario rechaza)
```

| Estado | Login Permitido | Acción Disponible |
|--------|----------------|-------------------|
| `PENDIENTE_APROBACION` | ❌ (mensaje de espera) | Operario: Aprobar / Rechazar |
| `ACTIVO` | ✅ | Normal |
| `RECHAZADO` | ❌ (mensaje rechazo) | Contactar administrador |

### Notas Técnicas

- **Migración Flyway:** Versión `V14` — el DEFAULT 'ACTIVO' asegura que los usuarios ya existentes no se ven afectados.
- **Enum en MySQL:** Requiere alterar la columna `rol` del ENUM a un nuevo ENUM `estado_cuenta`. Se agrega como columna separada, no se modifica `rol`.
- **Compatibilidad con HU-13 (móvil):** El mismo flujo aplica cuando el cliente se registra desde la app móvil — el backend rechazará el login hasta que sea aprobado.
- **No se envían emails en esta fase:** La notificación de aprobación será solo por el sistema de notificaciones del frontend (polling). Una futura HU podría agregar envío de emails.
