## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** MVVM (Model-View-ViewModel)
*   **Justificación:** Estándar de industria para Android. Desacopla la UI de la lógica de negocio y facilita el testing.
*   **Estructura de Paquetes:**
    `com.sena.proyecto`
    *   `data`: Capa de acceso a datos.
        *   `api`: Interfaces Retrofit (`ApiService`).
        *   `model`: Data Classes / DTOs.
        *   `repository`: Repositorios que abstraen la fuente de datos.
    *   `ui`: Capa de presentación (Activities, Fragments, ViewModels).
        *   `auth`: Login, Registro.
        *   `dashboard`: Pantalla principal.
    *   `utils`: Constantes, Extensiones, Helpers.

**Stack Tecnológico:**
*   **Lenguaje:** Kotlin
*   **Red:** Retrofit + Gson
*   **Concurrencia:** Kotlin Coroutines
*   **Carga de Imágenes:** Glide o Picasso (a definir en implementación).
*   **Inyección de Dependencias:** Manual (Container pattern simple) para evitar complejidad de Hilt en fase inicial.

### Especificaciones Técnicas

**Configuración de Red:**
*   **Base URL:** `http://10.0.2.2:8080/api/` (IP especial para emulador Android que apunta a localhost del host).
*   **Cliente HTTP:** OkHttp con logging interceptor (para debug).

**Refactorización Inicial:**
*   Mover `MainActivity` y `DashboardActivity` al paquete `ui`.
*   Crear paquete `data` vacío preparado para la integración.

## Contexto Técnico (SM)

### Base del Refinamiento
**Análisis arquitectónico utilizado:** SÍ
**Decisiones adoptadas:** MVVM, Retrofit, Coroutines, DI Manual.

### Análisis de Impacto
*   **Gestión de Dependencias:** El proyecto usa **Version Catalog** (`gradle/libs.versions.toml`). Las nuevas librerías deben registrarse allí primero.
*   **Permisos:** Se requiere agregar `android.permission.INTERNET` al `AndroidManifest.xml`.
*   **Seguridad Local:** Para desarrollo local (HTTP), se debe habilitar `android:usesCleartextTraffic="true"` en el manifiesto.
