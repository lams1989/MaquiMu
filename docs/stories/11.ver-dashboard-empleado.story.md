## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** CQRS (Query Side) - Agregación de Datos

**Justificación:**
*   **Eficiencia:** Permite realizar consultas optimizadas (`COUNT`) sin cargar entidades completas.
*   **Desacoplamiento:** La capa de aplicación orquesta la obtención de datos de múltiples dominios (Maquinaria, Alquiler, Facturación) sin acoplarlos entre sí.
*   **Escalabilidad:** Facilita la adición de nuevos KPIs en el futuro sin modificar los dominios base.

**Componentes Principales:**

*   **Aplicación (Consulta):**
    *   `ConsultaDashboardOperario`: DTO de solicitud (vacío o con filtros).
    *   `ManejadorDashboardOperario`: Orquestador que invoca los DAOs.
    *   `DashboardOperarioDTO`: Objeto de transferencia con los KPIs.

*   **Dominio (Puertos):**
    *   `MaquinariaDao`: Añadir `long contarPorEstado(EstadoMaquinaria)`.
    *   `AlquilerDao`: Añadir `long contarPorEstado(EstadoAlquiler)`.
    *   `FacturaDao`: Añadir `long contarPorEstado(EstadoPago)`.

*   **Infraestructura:**
    *   `ConsultaControladorDashboard`: Endpoint REST.
    *   Repositorios JPA implementando los conteos (`countBy...`).

### Especificaciones Técnicas

**API Endpoint:**
*   `GET /api/maquimu/v1/dashboard/operario`
*   **Respuesta:**
    ```json
    {
      "maquinariaDisponible": 15,
      "alquileresActivos": 8,
      "facturasPendientes": 3
    }
    ```

**Integraciones:**
*   El manejador agrega datos de 3 fuentes distintas.
*   Consultas de solo lectura optimizadas.
