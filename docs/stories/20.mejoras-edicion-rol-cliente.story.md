# Historia #20: Mejoras — Edición de Rol de Usuario desde el Módulo de Clientes

**Estado:** Finalizada ✅
**Fecha Creación:** 2026-04-14
**Fecha Implementación:** 2026-04-14
**Origen:** Solicitud directa (Mejora de funcionalidad)
**Estimación:** 3 Puntos

---

**Como:** Operario del sistema
**Quiero:** Poder editar el rol de usuario (CLIENTE / OPERARIO) desde el formulario de edición de cliente en el módulo de Clientes (`/admin/clients`).
**Para:** Gestionar los roles de los usuarios asociados a clientes sin necesidad de acceder directamente a la base de datos, facilitando la administración de permisos.

**Criterios de Aceptación:**

1.  **Selector de rol en edición de cliente:**
    *   Al editar un cliente que tiene usuario asociado (`usuarioId` no nulo), se muestra un selector de rol con las opciones ya definidas: `CLIENTE` y `OPERARIO`.
    *   El selector muestra el rol actual del usuario preseleccionado.
    *   Si el cliente no tiene usuario asociado (fue creado directamente por el operario), el selector de rol no se muestra.

2.  **Cambio de rol persistente:**
    *   Al guardar los cambios del cliente, si el rol fue modificado, se envía una petición adicional al backend para actualizar el rol del usuario.
    *   El cambio de rol y la actualización de datos del cliente se ejecutan en paralelo (forkJoin).
    *   Si alguna de las operaciones falla, se muestra un mensaje de error al operario.

3.  **Endpoint de cambio de rol:**
    *   `PATCH /api/maquimu/v1/usuarios/{id}/cambiar-rol` — cambia el rol del usuario. Solo accesible por OPERARIO.
    *   El endpoint valida que el rol enviado sea uno de los valores definidos en `RolUsuario`.

4.  **Endpoint de consulta de usuario:**
    *   `GET /api/maquimu/v1/usuarios/{id}` — retorna los datos del usuario (incluye `rol`). Usado para cargar el rol actual al abrir el modal de edición.

5.  **Seguridad:**
    *   Solo los operarios pueden cambiar roles.
    *   Los endpoints están protegidos por autenticación JWT.

---

## Análisis Arquitectónico

### Decisiones de Diseño

**Patrón:** CQRS — nuevo comando `CambiarRol` con su manejador correspondiente.

**Justificación:**
- Se agrega un método `cambiarRol(RolUsuario)` al modelo de dominio `Usuario`, siguiendo el patrón existente de métodos de transición.
- Se reutiliza la infraestructura de persistencia existente (`UsuarioRepositorio.guardar()`).
- En el frontend, se usa `forkJoin` para ejecutar la actualización del cliente y del rol en paralelo cuando ambos cambian.
